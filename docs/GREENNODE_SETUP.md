# GreenNode setup guide (for Alois — no prior experience assumed)

Written 2026-07-02. GreenNode's help articles are not machine-readable from outside,
so exact menu names may differ slightly — when in doubt, describe/screenshot the screen
and Claude adapts live. Official articles (readable in your browser once logged in):
- https://helpdesk.greennode.ai/portal/en/kb/articles/manage-a-notebook-instance
- https://helpdesk.greennode.ai/portal/en/kb/articles/connect-to-notebook-instances

## Part A — Account + money (only you can do this, ~10–15 min, one time)

0. FIRST: ask internally at VNG whether the company already has a GreenNode
   tenant/credits (GreenNode is VNG's cloud). If yes, skip to Part B with that account.
1. Go to https://register.greennode.ai/signup — sign up with your email, verify
   via the email link.
2. Log in. Find the **Billing / Payments / Top-up** section. GreenNode is prepaid:
   Visa/Mastercard or bank transfer. Start with a modest top-up (~$50–100) —
   enough for provisioning + the first training benchmark; scale up after we
   know real cost per dance. Reference: RTX 4090 list ≈ $610/month ≈ $0.85/hour.

## Part B — Create the notebook instance (~10 min, guided)

In the console, look for **AI Platform → Notebook instances → Create** (naming may vary).
Choices that matter:

| Setting | Pick | Why |
|---|---|---|
| GPU | 1× RTX 4090 | pinned in architecture |
| Region | nearest (VN/SEA) | latency + data transfer |
| Image / environment | PyTorch + CUDA 12.x if offered; plain Ubuntu 22.04 + CUDA also fine | Claude installs the rest |
| Disk / volume | ≥100 GB, ideally 150–200 GB, **persistent** | Isaac Lab + checkpoints are huge |
| Persistence on stop | keep data when stopped | training artifacts must survive |
| SSH key field (if present) | STOP and tell Claude | he'll generate a key for you to paste — best access mode |

Launch it and wait for status **Running**.

## Part C — Hand over access (~2 min)

1. Click **Open / Connect** on the instance → JupyterLab opens in a browser tab.
2. Copy the FULL URL from the address bar — it usually contains `?token=...`
   (that token is the secret that grants access).
3. If the URL has no token, the instance page usually shows connection info
   (URL + token, or an SSH command) — copy that instead.
4. Paste it to Claude in the project chat. From that moment everything is
   hands-off: provisioning GVHMR + Isaac Lab, benchmarking training on
   `data/dance1_subject2_seg.csv`, syncing artifacts back.

SECURITY: that URL/token = full control of the cloud machine. Don't post it
anywhere else; it can be regenerated by restarting the instance if leaked.

## Part D — Cost control

- **Stop** the instance whenever Claude says a phase is done — stopped instances
  usually only bill small storage fees, not GPU time.
- Long training runs (potentially overnight for 2–3 min dances) need it ON the
  whole time; Claude runs them under tmux/nohup so laptop reboots don't matter,
  and records job state in `logs/jobs.md`.
